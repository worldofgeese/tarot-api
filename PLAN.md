# Tarot API — Three-Tier Test Application

## Purpose
Test the full SDLC inner+outer loop: plan → TDD → dispatch → monitor → verify → PR.

## Architecture

### Tier 1: SQLite Database
- 78 tarot cards (22 Major Arcana + 56 Minor Arcana)
- Schema: `cards(id, name, arcana, suit, number, upright_meaning, reversed_meaning, keywords, image_desc)`
- Seed from a JSON file

### Tier 2: Bun/Elysia API
- `GET /api/cards` — list all cards (paginated, filterable by arcana/suit)
- `GET /api/cards/:id` — single card with full details
- `GET /api/spread/:type` — draw a spread (3-card, celtic-cross, single)
- `GET /api/search?q=` — FTS5 search across name, meanings, keywords
- JSON responses, proper error handling, CORS

### Tier 3: HTML Frontend (server-rendered, like goetia-dashboard)
- Landing page: grid of all 78 cards with SVG placeholder art
- Card detail page: full card info, upright/reversed meanings
- Spread page: interactive spread with drawn cards
- Search with live results

## Stack
- Runtime: Bun
- Framework: Elysia
- DB: bun:sqlite + FTS5
- Templates: tagged template literals (same pattern as goetia-dashboard)
- Tests: bun:test (unit + integration)
- E2E: Playwright via bun:test + setup.ts preload pattern

## Test Plan (TDD)

### Unit Tests (write FIRST)
1. Card data integrity: all 78 cards present, no nulls in required fields
2. Spread logic: 3-card returns exactly 3, no duplicates
3. Celtic cross returns exactly 10, no duplicates
4. Search: "death" returns Death card, "love" returns relevant cards
5. Pagination: limit/offset work correctly
6. Filter: arcana=major returns 22, suit=cups returns 14

### Integration Tests
7. API endpoints return correct status codes
8. API returns valid JSON with expected schema
9. Search endpoint handles empty query gracefully
10. Spread cards are randomized (run 10x, not all identical)

### E2E Tests (setup.ts preload pattern)
11. Landing page shows 78 card tiles
12. Card detail page renders all fields
13. Spread page draws cards on button click
14. Search returns results for known queries

## Constraints
- No external APIs or AI generation — all card data is static JSON
- Dark theme (deep purple/indigo — fits the tarot aesthetic)
- Mobile-responsive grid
- All tests must pass before any commit
- Follow TFH project template (README, scripts, Containerfile, CI)
- Use `bun run build-data` for seed, `bun run test` for unit+integration, `bun run e2e` for E2E
